# PRD 质量判据

## 唯一真源

本文件是 PRD 质量判据的**唯一真源**。作者自检用它、独立评审用它、CI lint 可以基于它写脚本。

所有 PRD 质量问题归类到两组：

- **6 维度内容质量**（下方"维度 1–6"）
- **P 类写作原则**（11 条硬约束，详见 `prd-writer.md`）

---

## 6 维度内容质量

### 维度 1 — 内容失焦

**应当检查**：每章节只谈自己的主题。

**需标出的模式**：
- UI 细节出现在"功能列表"章节（功能列表应只说"做什么"不说"怎么展示"）
- 跨模块业务规则混进单个功能的属性说明（跨模块规则应集中章节写）
- "原型图"章节承担业务规则说明（原型只负责视觉，规则在对应章节）

**不要误伤**：
- 在"业务规则"章节里引用字段名不是失焦，是必要
- 在"用户故事"里描述边界 case 不是失焦

### 维度 2 — 结构组织

**应当检查**：按类型分组，同类章节共享模板。

**需标出的模式**：
- 校验、输出、约束混在一串 bullet 里（应按固定结构分组）
- 同类功能（如"创建 X"和"创建 Y"）使用不同的章节结构
- 操作型功能没有"触发方式 → 字段表 → 成功/失败路径"三段式

**不要误伤**：
- 章节编号可不完全连续（允许跳号）
- 不同形态的 L2 功能用不同模板是必然

### 维度 3 — 顺序一致性

**应当检查**：同一组对象在文档所有位置保持相同的顺序。

**需标出的模式**：
- 字段顺序在"功能清单 / 架构图 / 功能详述 / 原型图"之间不一致
- Mermaid 节点排列与正文术语顺序相反
- 模块在目录里是 A/B/C，正文里是 B/C/A

**不要误伤**：
- 跨章节的对象可以按不同维度排序（按字母 vs 按业务优先级），只要每处排序依据明确

### 维度 4 — 歧义表述 ⚠️ 始终需要澄清

这是最容易翻车的维度。两种子类：

**4a. 技术术语泄漏**：PRD 里出现 HTTP 状态码、foreign key、SQL 片段、表名、列名。应转回业务语言。

**4b. 模糊业务词**：

需警惕的歧义词库：
- "模糊搜索" / "全文搜索" / "智能匹配"
- "自动同步" / "及时更新" / "实时刷新"
- "相关" / "合适" / "合理"
- "软删除" / "逻辑删除"
- "批量操作" / "批量处理"

**正确做法**：遇到这些词，必须把它具象化成"在 XXX 场景下，用户看到 YYY"。独立评审时要主动追问。

### 维度 5 — 前后自洽

**应当检查**：文档内部不自相矛盾。

**需标出的模式**：
- 第 3 章说"支持批量编辑"，第 4 章业务规则说"一次只能编辑 1 条"
- 用户故事说"用户可以删除"，Gherkin AC 只写了创建和编辑
- 追溯矩阵里某功能点找不到对应的指标

**不要误伤**：
- MVP 不做的能力和"未来规划"章节里的声明不算矛盾（只要分章节清楚）

### 维度 6 — 字段对齐 SOT（业务对象）

**应当检查**：PRD 中出现的字段名 100% 能在对应 `*-business-objects.md` 里找到。

**需标出的模式**：
- PRD 写"品牌名称"，SOT 写"品牌"（自造别名）
- 字段中英文混用（PRD 写 `brand_name`，SOT 写"品牌名称"）
- PRD 出现了 SOT 没定义的字段（要么 SOT 遗漏、要么 PRD 自创）

**不要误伤**：
- SOT 文件本身不存在时，本维度降级为"标出应当有权威定义但找不到真源的字段名"

## P 类原则自查（11 条）

这 11 条来自 `prd-writer.md`。自查时对每条打 pass/fail/risk：

| # | 原则 | 快速自查 |
|---|---|---|
| P1 | 三层指标（北极星/输入/护栏） | 三层齐全？每个指标有基线/目标/周期/数据源？ |
| P2 | INVEST 用户故事 | 每条故事过了 I-N-V-E-S-T 六项？ |
| P3 | Gherkin AC 覆盖三类场景 | 每条 AC 有 Happy/Edge/Error？ |
| P4 | 追溯矩阵闭环 | 矩阵存在？每行五列齐全？ |
| P5 | SOT 机械对齐 | 字段名与业务对象文档一致？ |
| P6 | 禁用技术术语 | 没有外键/SQL/状态码/表名？ |
| P7 | 按模块拆解 | 章节按业务模块而非页面组织？ |
| P8 | 两种功能详述结构 | 操作类用三段式？页面类用布局式？ |
| P9 | 字段表三合一 | 字段/必填/校验规则/校验失败提示 4 列？ |
| P10 | 校验时机分层 | 静态 vs 业务校验分别指明？ |
| P11 | 基础三要素 | 用户场景 / 成功指标 / 风险 三段齐全？ |

## 自查输出格式

用下面的 Markdown 表格输出：

```markdown
# PRD 自检报告：<PRD 文件名>

**检查时间**：YYYY-MM-DD
**PRD 行数**：N 行

## 维度总览

| 维度 | ✅ | ❌ | ⚠️ |
|---|---|---|---|
| 1. 内容失焦 | N | N | N |
| 2. 结构组织 | N | N | N |
| 3. 顺序一致性 | N | N | N |
| 4. 歧义表述 | N | N | N |
| 5. 前后自洽 | N | N | N |
| 6. 字段对齐 SOT | N | N | N |
| P 类原则（11 条） | N | N | N |

## ❌ 待修复项（按行号升序）

- **[第 XXX 行 / 章节 "X.X 小节名"]** 问题描述
  → 建议：一句话修改方向

## ⚠️ 待确认项

- **[第 XXX 行]** 风险描述
  → 请确认：具体问题

## 下一步建议

- 修完 ❌ 项再跑一次本检查
- 全绿后可走独立评审
```

## 自查 vs 独立评审的区别

| 场景 | 作者自查 | 独立评审 |
|---|---|---|
| 视角 | 作者自己 | 第三方 reviewer |
| 交互 | 无多轮对话，一次输出 | 可多轮澄清歧义 |
| 耗时 | 2–5 分钟 | 15–30 分钟 |
| 产出 | pass/fail 清单 | 结构化修改报告 |
| 触发 | 写完/改完立即跑 | 提审或里程碑评审 |

**关键**：两者用同一份判据（本文件），避免作者和 reviewer 打架。自查通过不代表免评审；自查是"不交废稿"，评审是"找出盲区"。

## 不做的事

- 不多轮对话（歧义词由作者自己判断，或交给独立评审）
- 不给"优化一下 XX 章节"这种无行号指令
- 不自动修 PRD（只标问题，不改文件）
- 不评判 PRD 是否"值得做"——超出本文件范围
