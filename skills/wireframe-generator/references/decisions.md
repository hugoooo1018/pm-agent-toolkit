# Decisions

PRD 不可能写清全部细节。按三类处理：**锁死** / **必问** / **自主**。

## 锁死（NEVER ASK）

这些决策已经和用户对齐过，不再询问：


| 决策                       | 规则                                                                                                      |
| ------------------------ | ------------------------------------------------------------------------------------------------------- |
| 页面范围                     | 只画主内容区。不画顶栏、侧栏、用户头像、全局通知。面包屑仅在 PRD 明确提到时才画                                                              |
| 语言                       | 中文为主，按钮/label/字段名全用 PRD 的中文术语                                                                           |
| PRD 未描述的字段               | 不加（不要无中生有"副标题"、"slogan"等）                                                                               |
| 空灰条                      | 是 bug。所有 `.wf-text` 必须有内嵌文字                                                                             |
| 必填标记                     | PRD 说必填的字段，`.form-label` 末尾加 `*`                                                                        |
| 外链按钮                     | 末尾加 `↗` 符号                                                                                              |
| 按钮顺序（弹窗）                 | 取消（outline）在左，主动作（实心）在右                                                                                 |
| JPG 保留                   | 永远不删 `docs/business/<module>/assets/<page>.jpg`                                                         |
| A/B/C/D 分类          | 见 `design-system.md`。动态字段（数据）用 `.wf-text`；静态 UI 文案（硬编码）用普通文字。判断法：后端看到会问"这是哪个 DB 字段"→ 动态；会说"这是前端硬编码"→ 静态 |
| 交互标记一律不用       | 不使用 `.click-chip` / `.attr-chip` / `.has-chip`。所有点击/跳转/触发/校验/弹窗等行为全部由 PRD 承担，wireframe 只负责结构和布局         |
| label / 板块注释（`.label`）触发范围 | 每个功能区（section / card / dialog）挂一张 `.label`，文案固定前缀 `板块：` + 板块名。列表内部的 item-card / 表格行等内层元素不重复挂            |
| 多项目编号                    | 列表/表格 N 个同类占位时，每个动态字段名后加 `1` ~ `N`（半角空格 + 阿拉伯数字）。表头行和单实例（弹窗、详情页顶部信息卡）不编号                                |


## 必问（BLOCKING）

这些情况**停下来问用户**，不能自主推断：

- **字段级歧义**：PRD 说"等"、"操作区"、"统计数据"但没列具体项
- **布局级歧义**：PRD 说"两栏"但没说左右放什么；"列表"没说卡片还是表格
- **资源级歧义**：找不到 PRD / PRD 无目标页章节 / 同一页在多个 PRD 都提到
- **契约缺失**：无 `docs/business/<module>/` 目录 / PRD 引用的 JPG 不存在

**询问方式**：聊天里用简洁 markdown 列表问，每次聚焦 1-3 个具体问题。不用 AskUserQuestion（这类歧义是 long-tail 不适合预设选项）。

## 自主（DECIDE + COMMENT）

自行推断，决策记在 HTML 顶部 `<!-- Autonomous decisions: ... -->` 注释里。


| 情况        | 默认                                               |
| --------- | ------------------------------------------------ |
| 列表卡片数量    | **按 PRD**；PRD 未指定时 3-5 之间视密度选                    |
| 表格数据行数    | 4                                                |
| 表单字段顺序    | 跟随 PRD                                           |
| 区域间距      | `16px` ~ `24px` 视密度                              |
| 弹窗宽度      | 表单 480px / 确认对话框 400px                           |
| 表格列数      | 按 PRD 提到的列平均分配 grid-template-columns；操作列固定 120px |
| **搜索框占位** | 按 PRD 自由描述（如"按商品名称/编号搜索"、"搜索框"等均可）。固定词不再硬性要求     |
| 确认对话框提示   | PRD 有原文直接用；没有则套"确认<动作><对象>'{变量}'？此操作不可恢复。"       |
| 响应式       | 768px 断点由 `_wireframe.css` 默认规则处理                |


## 决策边界的判断


| 歧义影响                     | 归类  |
| ------------------------ | --- |
| 后端字段/类型                  | 必问  |
| 用户看到的布局                  | 必问  |
| 视觉细节（间距、圆角、卡片数）          | 自主  |
| 排版/占位长度（short vs medium） | 自主  |


**Fallback**：拿不准偏向问。澄清成本 < 返工成本。

## 避免过度推断

- 不在 PRD 没提到的位置加"用户头像"、"通知按钮"、"搜索图标"等管理后台常见元素
- 不把 PRD 描述延伸解读成"应该有"某字段（如 PRD 说"显示商品"就加"商品图片"）
- 不自动加 empty/loading/error 三状态（PRD 未要求就不做）
