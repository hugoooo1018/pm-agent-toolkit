# 正面例子 — 符合 checklist 的写法

此文件为 `prd-review` skill 提供改写建议的风格参照。Stage 3 起草「建议改写」时，读对应维度小节并模仿其语气——让建议听起来像项目自己的声音，不是通用模板。

每条例子包含三个机器可用字段：
- **Input** — 一段符合该维度通过标准的 PRD 文本
- **Why it passes** — 使其合规的具体信号
- **Reusable snippet** — 可套用到建议中的句式模板

## 维度 1 — 内容失焦

### Example 1 — 干净的字段表

**Input:**

| 字段 | 必填 | 校验 | 错误提示 |
|---|---|---|---|
| **Rule 名称** | 是 | 非空；同一 Strategy 下唯一 | "必填" / "名称已存在" |
| **描述** | 否 | 最长 200 字符 | "超出长度" |

**Why it passes:**
- 字段表只携带业务属性（必填、校验、错误提示）
- 无 UI 词汇（hover、胶囊、icon）
- 无跨模块行为（"Strategy 如何引用 Rule"这种描述）

**Reusable snippet:**
"字段表只包含 `字段`、`必填`、`校验`、`错误提示` 四列。交互/UI 细节（tooltip 文本、hover 态、icon 位置）移到原型图说明。"

### Example 2 — 原型图说明只描述可见元素

**Input（Rule 列表页截图下方说明）:**
> 列表展示 5 列（名称、状态、创建人、创建时间、操作）。操作列中内联按钮：编辑、停用、删除。

**Why it passes:**
- 说明只描述可见元素
- 没把业务规则藏在说明里

**Reusable snippet:**
"原型图说明：'<页面> 显示 <列 / 分块>。<控件> 呈现为 <可视形式>'。业务规则放别处，必要时用 `详见业务规则 §X` 指向。"

## 维度 2 — 结构组织

### Example 1 — 分组清晰的功能描述

**Input:**

**创建 Rule**
- **触发**：点击 Rule 列表页 "新建 Rule"，打开创建对话框。
- **自动填充**：创建人 = 当前用户；创建时间 = 服务器时间。

**表单字段：**

| 字段 | 必填 | 校验 | 错误提示 |
|---|---|---|---|
| **Rule 名称** | 是 | 非空；同一 Strategy 下唯一 | "必填" / "名称已存在" |
| **描述** | 否 | 最长 200 字符 | "超出长度" |

- **成功后**：对话框关闭，列表刷新，toast "Rule 创建成功"。
- **失败后**：对话框保持打开，错误信息显示在对应字段下方。

**Why it passes:**
- 触发 / 自动填充 / 字段 / 结果，每块独立
- 校验和错误提示在同一字段行里，不散开
- 同类功能（编辑 Rule、删除 Rule）用完全一致的结构

**Reusable snippet:**
"Action feature 模板：触发 → 自动填充 → 字段表 → 成功后 → 失败后。同类功能（创建/编辑/删除）结构保持一致。"

## 维度 3 — 顺序一致性

### Example 1 — 同实体字段在列表和详情顺序一致

**Input:**
- Rule 列表列：**名称 / 状态 / 创建人 / 创建时间**
- Rule 详情块：**名称 / 状态 / 创建人 / 创建时间**
- Rule 编辑表单：**名称 / 状态 / 创建人 / 创建时间**（置灰字段不可编辑）

**Why it passes:**
- 同一实体在所有视图使用同一基准顺序
- 基准遵循业务主流程（先身份、再状态、再 owner、再时间戳）

**Reusable snippet:**
"每个实体确立一个基准顺序（建议：业务主流程；名称 → 状态 → owner → 时间戳）。列表、详情、表单三视图都遵循该基准。"

### Example 2 — 同类页面操作按钮顺序一致

**Input:**
- Rule 列表操作：**编辑 / 停用 / 删除**
- Strategy 列表操作：**编辑 / 停用 / 删除**
- Customer 列表操作：**编辑 / 停用 / 删除**

**Why it passes:**
- 同类列表页使用同一操作顺序
- 约定：安全操作在左，破坏性操作在右

**Reusable snippet:**
"同类列表页共用同一操作顺序。约定：非破坏性 → 可恢复 → 破坏性（编辑 → 停用 → 删除）。"

## 维度 4 — 歧义表述

### Example 1 — 具象化的匹配规则

**Input:**
> Rule 名称搜索：前缀匹配，不区分大小写，最多返回 50 条。

**Why it passes:**
- 匹配语义明确（前缀、不区分大小写）
- 条数上限写明
- 研发可以直接实现，不用再问

**Reusable snippet:**
"`<字段> 搜索：<前缀 | 子串包含 | 分词> 匹配，<区分 | 不区分> 大小写，最多返回 <N> 条。`"

### Example 2 — 定时同步 + 失败处理

**Input:**
> Meta 广告账户数据每 30 分钟轮询一次。Meta API 返回错误时，保留上一次成功的快照，并在记录上标注「Stale since [时间戳]」。下一次成功拉取后清除该标记。

**Why it passes:**
- 时机明确（每 30 分钟）
- 失败行为定义清楚（保留快照 + 加标记）
- 恢复行为定义清楚（下一次成功拉取清除）

**Reusable snippet:**
"`<数据源>` 每 `<周期>` 拉取一次。失败时 `<保留 | 丢弃 | 回退>`，`<用户可见状态>`。恢复：`<清除该状态的条件>`。"

### Example 3 — 技术术语 + 业务注释

**Input:**
> 停用 Rule 是幂等操作（对已停用的 Rule 重复调用停用不产生额外影响）。

**Why it passes:**
- 术语保留（确实精准）但括号里给了业务读者看得懂的解释
- 非技术干系人也能理解效果

**Reusable snippet:**
"`<技术术语>`（`<面向业务读者的白话解释>`）。"

## 维度 5 — 前后自洽

### Example 1 — 全局规则只定义一次

**Input（业务规则章节）:**
> 所有列表页的删除在调用后端前需二次确认弹窗。

**Input（每个功能章节）:**
> 删除操作：详见业务规则 §2。

**Why it passes:**
- 规则定义一次
- 使用它的每个功能用链接指回，而不是复述

**Reusable snippet:**
"`全局规则在业务规则 §<X> 定义一次。每个使用它的功能写：详见业务规则 §X。`"

### Example 2 — 原型与功能点双向对齐

**Input:**
- 功能列表：查看 Rule 详情、编辑 Rule、停用 Rule、删除 Rule
- 原型图册：Rule 列表页、Rule 详情页、Rule 编辑对话框、停用确认、删除确认

**Why it passes:**
- 每个功能点都有对应原型
- 每个原型页都有对应功能
- 两边都没孤儿

**Reusable snippet:**
"`每个功能点列出实现它的原型；每个原型页列出它服务的功能。两边出现孤儿，要么补另一半，要么删孤儿。`"

### Example 3 — 指标公式只定义一次

**Input:**
> Rule 命中率 = `命中次数 / 评估次数`（除以零返回 0；跳过评估计入分母。详见数据口径 §3.2）。

**Why it passes:**
- 公式只写一次
- 边界条件指向权威口径章节
- 不会再有第二个公式

**Reusable snippet:**
"`<指标> = <公式>。详见数据口径 §<X>。`"

## 维度 6 — 字段对齐

### Example 1 — PRD 逐字使用权威名字

**Input（一份评审干净的 PRD 摘录）:**
> **品牌**实体的属性：**品牌名称**（必填，唯一）、**品牌 URL**（必填）、**商品数量**（系统维护）。

**Why it passes:**
- 字段名与 `customer-business-objects.md` 完全一致
- 首次出现时 bold，让领域术语视觉上有区分（参 PRD Writer Rule P5 SOT 对齐要求）
- 无自创别名

**Reusable snippet:**
"`引入任何字段前，先在同域 *-business-objects.md 里搜权威名字。逐字复制（casing、中英文混写都照搬）。首次出现用 ** 加粗。`"

### Example 2 — 新字段先入 business-objects

**Input（更新顺序）:**
> 1. 先更新 `rule-business-objects.md`，添加 `priority`：integer、可选、1–100、默认 50、用途为决定同一 Strategy 内评估顺序。
> 2. 再更新 PRD 引用 priority，规格与 business-objects 一致。
> 3. 评审时 diff 两份文档确认无漂移。

**Why it passes:**
- business-objects 文档作为唯一真源先更新
- PRD 从它派生，天然对齐
- 评审有可验证的对齐步骤

**Reusable snippet:**
"`新增字段时：先在 business-objects 写完整规格（类型、必填、约束、用途）。再在 PRD 引用权威名字。评审 diff 两份确认对齐。`"
