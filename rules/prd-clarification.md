# PRD 需求澄清

## 定位

PRD 写作前的必经过滤器。把"领导扔来的一句话"或"PM 脑子里的模糊想法"转化成结构化的**需求理解总结**，作为后续 PRD 正文写作的输入。

适用：需求来源模糊、缺场景、缺指标、缺边界的任意 PRD 写作前置。
不适用：已有成熟需求文档只需格式化的场景。

## 基础原则

- **P1 三层指标**：澄清阶段至少要挖出北极星指标候选和数据源
- **P5 SOT 对齐**：澄清第一件事就是确认是否存在业务对象文档
- **三要素齐全**：澄清结束时必须能给出用户场景 / 成功指标 / 风险三段

详见 `prd-writer.md`。

## 第 0 步：先问 SOT 与上游信号

收到需求描述后，用下面这段开场（不要直接冲进 5 维度提问）：

> "进入澄清前先确认几件事：
> 1. 这个功能的业务领域是否已有业务对象文档（`*-business-objects.md`）？如有请告知路径。
> 2. 有没有已立项的北极星指标或 OKR 需要对齐？
> 3. 有没有用户研究、竞品调研或已有产品可参考？"

得到回答（哪怕是"都没有"）后才进入 5 维度澄清。

## 第 1 步：从 5 个维度评估信息完备度

| 维度 | 关注点 |
|---|---|
| **背景与目标** | 为什么做？解决什么问题？北极星指标是什么？输入/护栏指标怎么定？ |
| **用户与边界** | 给谁用？核心场景是什么？哪些明确不做？ |
| **核心逻辑与数据** | 关键业务规则？涉及哪些数据和状态流转？异常怎么处理？ |
| **约束与依赖** | 技术限制、时间压力、合规要求？与现有系统怎么集成？ |
| **优先级与参考** | 是否分期交付？有没有竞品或内部产品可参考？ |

## 第 2 步：多轮澄清对话

- **硬约束**：至少完成 2 轮澄清对话才能判定是否可以收尾
- 优先问用户描述里**完全没覆盖**的维度
- 每轮 2–5 个问题，建议 2–3 轮，最多 4 轮
- 每轮给选项形式（a/b/c/d + "其他"），不要让用户写命题作文
- 用户说"差不多了""进入下一步"时，不要再凑轮次，直接进准入校验

## 第 3 步：准入 checklist

全部通过才算澄清完毕：

| ✓ | 检查项 |
|---|---|
| □ | **背景与目标**：已明确核心问题、目标用户痛点、至少 1 个北极星指标候选（可 `[待确认]`） |
| □ | **用户与边界**：已明确目标用户角色、至少 2 个核心使用场景、已知的 scope 边界 |
| □ | **核心逻辑与数据**：已明确关键业务规则或状态流转（可有 `[假设]`，但不能完全空白） |
| □ | **约束与依赖**：已了解主要技术约束或集成需求 |
| □ | **优先级与参考**：已明确是否分期交付、是否有竞品参考 |
| □ | **SOT**：已确认业务对象文档是否存在及其路径 |
| □ | **指标数据源**：至少能判断北极星指标是否有数据埋点支撑（无则 `[待数据团队确认]`） |

任何一项没过，针对缺失项再追一轮（总轮次不超过 4 轮）。用户未答或答模糊的，自行做专业判断并标注 `[假设]`。

## 第 4 步：输出需求理解总结

全部 checklist 通过后（或用户要求跳过），输出以下结构（逐字照用格式，不要换形式）：

```markdown
**📋 需求理解总结**

**一、需求理解**
- 背景与问题：{一段话概括}
- 目标用户：{核心角色及关键诉求}
- 核心 Outcome：{功能成功后用户/业务会发生什么可观测的改变——用户视角，非功能列表}
- 核心场景：{2-3 个最重要的使用场景}
- 关键约束：{技术/时间/合规}
- 不做的事项：{明确排除的内容}
- SOT 对齐：{业务对象文档路径 或 "无 SOT"}

**二、指标体系（候选）**
- 北极星指标：{名称 / 基线 / 目标 / 测量周期 / 数据源 或 [待确认]}
- 输入指标（2-4 个）：{同上格式}
- 护栏指标（1-3 个）：{同上格式}

**三、功能架构总览**
- 模块 1：{名称} — {一句话描述}
- 模块 2：{名称} — {一句话描述}

**模块间关系：**
- {A} 依赖 {B}（原因：...）

**标注为 [假设] 的部分：**
- [假设 1]：...（原因：...）

以上是我对需求的理解，请确认是否准确。
```

**不继续写 PRD 正文**——澄清的职责到此为止。用户确认后再进入正式写作。

## 跳过规则

若用户主动说"不用澄清直接写"：

- 至少确认"背景与目标"维度（核心问题 + 目标用户 + 北极星指标候选）
- 其余维度基于专业判断填，全部标 `[假设]`
- 仍然输出一份"需求理解总结"，让用户一眼看出哪些是假设

不要在用户没开口的情况下主动跳过。

## 示例

**输入**（一句话需求）：
> 我想做一个每日签到功能让用户每天打开 App

**第 0 步提问**：
> 进入澄清前先确认几件事：1) 这个功能的业务领域是否已有业务对象文档？2) 北极星指标目前锁定在哪个（DAU / 留存 / 时长 / 收入）？3) 有没有参考的竞品签到实现？

**第 1 轮澄清提问**（示例）：
```
1. 这个签到功能主要服务的用户是？
   a) 全体新老用户
   b) 仅新用户（提升次周留存）
   c) 仅沉默老用户（召回）
   d) 其他

2. 连续签到如果中断，怎么处理？
   a) 清零重来
   b) 保留进度，补签需付费
   c) 每周给一次免费补签
   d) 其他
```

**第 4 步输出**：一份填好的"需求理解总结"（见上文格式）。
