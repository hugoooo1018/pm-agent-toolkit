# PRD Review Checklist

---

## 1. 内容失焦

**问题定义：** 一个章节里出现了与它主题无关的内容。

### 具体表现形式

**① 把 UI 呈现混入对象说明**

- 表单字段说明中混入 UI 呈现细节：”只读胶囊标签“、”渲染时机“、”hover 态出现删除图标“等 UI 呈现细节出现在业务规则内的表单字段。

**② 把业务规则混入非业务规则章节**

- 属性列表中混入业务规则：对 Strategy 列表的描述"Rule 下包含的 Strategy，按创建时间升序评估，命中即停"，其中"按创建时间升序评估，命中即停"属于业务规则，不应出现在属性列表；
- 功能点描述中夹带其他模块的规则：在"创建 Rule"里具体描述"Strategy 如何引用 Rule"；
- 原型图章节反向承担业务规则说明：在 Rule 列表页原型图下直接写"命中即停，不再评估其他 Strategy"这类规则说明，代替业务规则章节应承担的职责。

### 通过标准（Pass）

- 每个章节描述只围绕它的主题展开，不混入其他主题的内容；

---

## 2. 结构组织混乱

**问题定义：** 同一区块内的信息未按类型或层次分组，或同一类章节结构不统一。

### 具体表现形式

**① 信息平铺堆放，未按类型或层次分组**

- 多类附属信息混在同一串 bullet：补充说明、提交结果（成功/失败）、约束条件、异常处理平铺罗列，未按类型分组；
- 输入侧校验与输出侧反馈交错排列：同一功能点下先写"手机号必填"（校验），再写"提交成功后跳转详情页"（反馈），紧接着又写"邮箱格式校验"（校验），校验与反馈穿插无序，读者需反复前后扫视才能拼接完整行为；
- 流程与例外分支未分层："前置条件 / 主流程 / 例外分支"平铺，主流程被例外说明淹没。

**② 同一类章节结构不统一**

- 同类章节使用不一致的结构模板：多段同属"页面功能描述"，一段先列字段再讲交互、另一段先讲交互再列字段，bullet 的层级、分组和顺序不对齐，读者无法按同一套习惯扫读。

### 通过标准（Pass）

- 同一区块内的信息按类型或层次分组，每组用小标题或加粗分隔（**按类型**：「补充说明 / 提交结果 / 约束条件 / 异常处理」归类，字段级信息以表格呈现；**按层次**：前置条件 / 主流程 / 例外分支分别成段）；
- 同类章节采用一致的结构，bullet 的层级、分组和顺序对齐，读者可按同一套习惯扫读。

---

## 3. 顺序不一致

**问题定义：** 同一组对象在文档多处出现时，排列顺序不一致。

### 具体表现形式

**① 模块 / 状态 在不同章节顺序不一**

- 模块在不同章节顺序不一：功能详述按 A→B→C→D 介绍模块，原型图却按 C→A→D→B 展示；
- 状态在多章节顺序不一：Rule 状态在业务规则章节、功能描述、原型图中排列顺序三处不同。

**② 字段 在同一实体的不同视图顺序不一**

- 同一实体不同表单字段顺序不同："创建表单"与"编辑表单"字段顺序不一致，读者无法直观对照差异；
- 列表页字段列与详情页字段块顺序不一致：Rule 列表页按"名称 / 状态 / 创建人 / 创建时间"排列，但 Rule 详情页按"名称 / 创建时间 / 状态 / 创建人"展示，同一实体两种视图字段排列不对应。

**③ UI 控件 在同类页面顺序不一**

- 筛选器、操作按钮顺序在同类页面未统一：Rule 列表页筛选器按"状态 / 创建人 / 创建时间"排列，Strategy 列表页却按"创建时间 / 状态 / 创建人"排列；同时 Rule 列表页操作按钮为"编辑 / 停用 / 删除"，Strategy 列表页却是"停用 / 编辑 / 删除"，同类页面 UI 控件排布不对应。

### 通过标准（Pass）

- 同一组对象在全文所有出现位置（跨章节、跨视图、跨页面）使用同一顺序；
- 基准顺序的选择规则：模块 / 字段按业务主流程顺序；状态 / 枚举按业务流转顺序；UI 控件按同类页面的约定（"保存"始终在"取消"右侧）。

## 4. 歧义表述

**问题定义：** 出现可能引起研发或业务部门产生歧义的词语。

### 具体表现形式

**① 技术术语泄露到业务描述中**

- 使用涉及删除语义的技术术语："硬删除 / 软删除 / 级联删除"，业务侧无法判断数据是否可恢复、是否连带删除；
- 使用涉及并发/事务的技术术语："幂等 / 原子操作 / 事务回滚"，业务读者无法判断对自己操作的实际影响；
- 使用涉及加载/缓存机制的技术术语："缓存穿透 / 懒加载 / 预加载"，指向技术实现机制，业务部门不应被要求理解。

**② 业务术语表述模糊**

- 使用未定义匹配方式的词语："模糊搜索"未说明是前缀 / 包含 / 分词 / 拼音；
- 使用未定义时机或频率的词语："自动同步"未说明同步时机、频率、冲突处理；
- 使用主观判断词："相关 / 类似 / 合适"无法映射到可实现的判定逻辑。

### 通过标准（Pass）

- 用业务语言表达行为和结果，替代技术术语（用"删除后无法恢复"替代"硬删除"）；若必须使用专业术语，紧跟括号给出业务含义（"幂等（重复提交不会产生额外影响）"）；
- 模糊词必须具象化为可判定的规则（"模糊搜索"→"对名称字段做前缀匹配，不区分大小写"）。

## 5. 前后行文不自洽

**问题定义：** 文档内部描述之间存在矛盾或断裂。

### 具体表现形式

**① 行为描述与业务规则矛盾**

- 流程描述与业务规则直接冲突：流程写"命中后进入下一条 Rule 继续匹配"，业务规则却规定"每次仅执行命中的单条 Rule"；
- 同一行为两处表述相反：一处说"停用后历史引用保留"，另一处说"停用将清空历史引用"；
- 字段属性与流程分支矛盾：字段标注"必填"，但提交流程中存在可跳过该字段的分支。

**② 原型图与功能详述无强映射关系**

- 原型图有页面但功能详述未覆盖：原型图中存在 Rule 详情页，但功能模块中无"查看详情"功能点；
- 功能点未落到原型图：功能点描述了"批量导出"，但原型图中无对应入口；
- 原型元素未在字段说明中列出：原型图展示了某筛选器，但字段说明未列出该筛选维度。

**③ 引用的页面 / 入口 / 术语在文档中无任何定义**

- 引用的页面无定义：触发方式提到"从 Strategy 详情页跳转"，但文档任何章节都未定义 Strategy 详情页；
- 引用的入口无定义：提到"通过右上角菜单进入设置"，但原型图中该菜单不存在。

**④ 数据口径前后不一致**

- 同一指标两处公式不同：Rule 命中率在业务规则章节写成"命中次数 / 评估次数"，在数据口径章节写成"命中次数 /（评估次数 - 跳过次数）"，两处分母不一致；
- 同一字段命名不一致：同一字段"规则 ID"在接口定义中写作 ruleId，在数据库表中写作 rule_code，命名规范未对齐。

**⑤ 同一规则在多处重复展开**

- 全局规则在多处重复展开：业务规则章节已定义"所有列表删除需二次确认"，但创建 Rule、创建 Strategy、编辑 Rule 三个功能描述里又各自复述了一遍二次确认逻辑。

### 通过标准（Pass）

- 文档内部无矛盾：行为描述、字段命名、指标口径、状态枚举值在全文保持一致，冲突时以业务规则章节为准；
- 所有引用（页面、入口、术语）在文档中均有明确定义；原型图与功能详述双向对应，不出现孤页或未落图的功能点；
- 全局规则只在定义处展开一次，其他位置以"详见 XXX"指向，避免未来改漏。
