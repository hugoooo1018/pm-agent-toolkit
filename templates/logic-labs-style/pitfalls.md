# 常踩坑清单

审查 PRD 时可逐条扫描；每条都给出"反例 → 正解"的对照。

## 结构性坑

### 1. 创建与编辑合并写

❌ 反例：
```markdown
**创建 / 编辑 Rule**
- 触发方式：在列表页点 Create Rule 打开弹窗，或在详情页点 Edit 打开弹窗
```

✅ 正解：拆成两个独立功能点，各自一份字段表。编辑型的字段表可引用 "同「创建 Rule」弹窗表单字段"，但结果状态必须分开写。

### 2. 把"不支持 XX"藏在段落中

❌ 反例：
```markdown
**删除 Rule**
- 约束与边界：
  - 业务边界：仅支持单条删除，暂不支持批量删除
```

✅ 正解：独立成小节
```markdown
**不支持批量删除**

MVP 阶段仅支持单条删除；批量删除能力放到第七章未来规划。
```

### 3. 跨模块规则只在 L1 概述里一笔带过

❌ 反例：只在「规则管理模块」概述里写一句"规则名称在同一广告主下唯一"。

✅ 正解：同时出现在第四章业务规则表（编号 R-01），L1 概述只"详见 R-01"。

---

## 「约束与边界」坑

### 4. 塞进超过两类内容

❌ 反例：
```markdown
- 约束与边界：
  - 业务边界：仅支持规则名称搜索
  - 数据影响：无
  - 前置条件：规则必须为未生效状态
  - 外部依赖：依赖 Meta Marketing API
  - 容错规则：API 失败时页面提示错误
```

✅ 正解：
```markdown
- 触发方式：...
  - 前置条件：规则必须为未生效状态    ← 上提到触发方式下
- 约束与边界:
  - 业务边界：仅支持规则名称搜索        ← 只留这一条，其他两类外迁
# 外部依赖 → 第七章 · 外部依赖表
# 容错规则 → 第七章 · 主要风险表的"应对措施"列
```

### 5. 空槽占位

❌ 反例：`数据影响：无` / `业务边界：无` / `约束与边界：无` 全是占位。

✅ 正解：没有就整段删除，不写"约束与边界"小节。

### 6. 前置条件双写

❌ 反例：触发方式下写了前置条件，约束与边界里又写一遍。

✅ 正解：只在触发方式下写，**约束与边界不重复**。

## 写作一致性坑

### 7. Mermaid 与正文说法不一致

❌ 反例：
- Mermaid 节点：`按创建时间升序`
- 正文描述：`按执行顺序升序排列`

✅ 正解：固定一种说法（例如统一为 `按执行顺序升序`），两处改一致。

### 8. 专有名词加粗不统一

❌ 反例：
- 首次出现：**规则状态 Toggle 开关**
- 再次出现：规则状态开关 / **规则状态** Toggle / toggle

✅ 正解：全文每次都写 **规则状态 Toggle 开关**，加粗 + 文本完全一致。

### 9. 字段顺序在各章节之间不对齐

❌ 反例：
- 功能需求清单：规则名称 → Ad Account → 作用层级
- 创建弹窗字段表：规则名称 → 作用层级 → Ad Account
- 原型图：作用层级 → Ad Account → 规则名称

✅ 正解：整份 PRD 固定一个字段顺序，贯穿四处：功能清单 / 架构图 / 功能详述 / 原型图。

## 字段表坑

### 10. 只写规则不给提示文案

❌ 反例：`| **规则名称** | 是 | 不为空；唯一 | |`（最后一列空）

✅ 正解：`| **规则名称** | 是 | 不为空；同一 **广告主** 下不允许重复 | "此字段为必填项" / "该规则名称已存在" |`

### 11. 业务校验没标作用域

❌ 反例：`唯一性校验` / `不允许重复`

✅ 正解：必须标作用域，例："同一 **广告主** 下不允许与其他 **Rule** 的 **规则名称** 重复"。

### 12. 系统自动生成字段单独列"系统自动处理"段

❌ 反例：
```markdown
- 提交结果状态：
  - 成功：弹窗关闭
- 系统自动处理：生成规则 ID、创建时间，状态默认"未生效"
```

✅ 正解：并入「成功」分支
```markdown
- 提交结果状态：
  - 成功：弹窗关闭；系统生成 **规则 ID** 和 **创建时间**；**规则状态** 默认"未生效"
```

## 页面型坑

### 13. 列表不写空状态

❌ 反例：主体区域只列出表格列，不提空状态。

✅ 正解：每个列表 / 表格都必须紧跟一条 `空状态：{描述}`，文案英文原文直引。

### 14. 空状态用中文"暂无数据"

❌ 反例：`空状态：暂无数据`

✅ 正解：英文原文直引，例：`No rules yet. Click "Create Rule" to create your first rule.`

### 15. 把操作细节在页面布局里展开

❌ 反例：列表行内"Delete"按钮展开写触发方式、确认弹窗、提交结果。

✅ 正解：只写"操作：'Delete' 按钮，详见「删除 Rule」"，细节在对应 L2 操作型功能里写。

---

## 执行型坑

### 16. 无 mermaid 流程图

❌ 反例：只有 1~4 点步骤，没配图。

✅ 正解：执行型**必须**配 `flowchart LR` + `subgraph` 流程图。

### 17. 异常分支外迁

❌ 反例：执行规则只写happy path，把异常归类到另一个"容错规则"小节。

✅ 正解：异常分支作为对应步骤的子 bullet 直接列出。

### 18. 提交结果状态列运行中异常

❌ 反例：
```markdown
- 提交结果状态：
  - 成功：...
  - 某条 Target Object 失败：跳过
  - API 超时：重试
  - 鉴权失败：终止
```

✅ 正解：只保留两个分支（任务启动成功 / 任务启动失败），运行中的异常已在执行规则步骤里说清。

## 路径 / 引用坑

### 19. 绝对路径

❌ 反例：`![原型图](/Users/hugo/Desktop/Product Manager/.../rule-list.jpg)`

✅ 正解：`![Rule 列表页](./assets/rule-list.jpg)`

### 20. "详见 XXX" 指向的小节不存在

❌ 反例：正文写 "详见业务规则 **批量删除规则**"，但第四章没这条规则。

✅ 正解：引用前先确认目标存在；不存在就在第四章补上或改引用目标。

## MVP 优先级坑

### 21. MVP 文档出现 P2

❌ 反例：功能需求清单里标记一条功能为 P2。

✅ 正解：MVP 文档只出现 P0 / P1。P2 的能力统一塞进第七章未来规划，以"列表 + 简述"形式列出，不进清单。
