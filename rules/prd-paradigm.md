# PRD「模块-功能-具体描述」范式

## 定位

本 Rule 定义**如何写 PRD 的章节内容**的规范。是一份**opinionated** 的范式，基于作者个人实践沉淀，适合需要严格结构化、多人协作、长期维护的 PRD 场景。

对应的**骨架模板**在 `../templates/default-style/`，可直接复制填写。

**不适合本范式的场景**：
- 战略级 PRD（产品愿景、市场论证）
- 一页纸 PRD / OPRD / PR-FAQ 等轻量格式
- 仅供内部脑暴的 draft

这种时候 fork 本仓库自建风格模板。

## 三层结构

所有模块级 PRD 严格三层：

- **L1 模块**：顶层业务域（如"规则管理"）
- **L2 功能**：模块内的操作或页面，分三种形态：
  - **操作型** — 创建 / 编辑 / 删除 / 状态切换 / 搜索
  - **页面型** — 列表页 / 详情页
  - **执行型** — 手动执行 / 批处理 / 循环任务
- **L3 字段细节**：每个字段必须含中文名、必填、静态校验、业务校验（标作用域）、校验失败文案

## 7 章固定骨架（模块级）

顺序不可调，编号不可变。可直接复制 `../templates/default-style/module-prd-7-chapter.md`。

```
一、产品背景与目标
  ├─ 背景与问题陈述
  └─ 产品目标（≤3 条可量化）

二、用户故事与功能需求
  ├─ 目标用户画像
  ├─ 用户故事（作为 X，我希望 Y，以便 Z；按模块分组）
  └─ 功能需求清单表 │ 优先级 │ 功能名称 │ 功能简述 │

三、功能模块详述
  ├─ 功能架构总览（graph TD + subgraph 分模块 + style 上色）
  └─ 各模块按 L1 模板展开

四、业务规则
  跨模块约束表 │ 编号 │ 规则描述 │ 适用范围 │

五、原型图
  每张图：页面说明 + `./assets/xxx.jpg` 引用

六、业务对象梳理
  指向独立文件 `./{module}-business-objects.md`

七、风险与依赖
  ├─ 主要风险表 │ 风险 │ 影响 │ 应对措施 │
  ├─ 外部依赖表 │ 依赖 │ 说明 │
  └─ 未来规划（可选；MVP 外的 P2 放此处）
```

## 10 章骨架（功能级）

功能级 PRD（单一功能，非模块）用 10 章骨架。详见 `../templates/default-style/feature-prd-10-chapter.md`。

关键差异：

- 加入显式的**指标体系章节**（三层指标 + 四要素）
- 加入显式的**验收标准章节**（Gherkin）
- 加入显式的**追溯矩阵章节**
- 功能模块详述章节内嵌 L1/L2/L3 三层结构

## L1 模块层模板

每个 L1 模块按以下结构展开：

```markdown
### {模块名称}模块

#### 模块概述
{3-5 句，依次说明：
 1) 在业务链路中的定位（父级 / 子级是谁）
 2) 与兄弟模块的层级关系
 3) 操作入口在哪里}

#### 功能详述
{逐个 L2 功能展开；按对应形态套模板}

#### 与其他模块的关系
- 依赖：{上游模块}
- 被依赖：{下游模块；MVP 无下游写"MVP 阶段无下游依赖"}
- 数据流向：{关键外键关系}
```

## L2 三形态模板

根据功能形态选一：

- **操作型** → `../templates/default-style/L2-operational.md`
- **页面型** → `../templates/default-style/L2-page.md`
- **执行型** → `../templates/default-style/L2-execution.md`

## 硬规则：Always do（每份 PRD 强制）

- 7 章骨架完整，章节编号与顺序不可调
- 每个 L2 功能按形态套用对应完整模板
- 写操作三件套齐全：触发方式 + 字段表（或引用） + 提交结果状态
- 列表组件必须给空状态（标题 + 描述）
- 核心业务术语全文加粗（如 **广告主** / **规则名称** / **规则状态 Toggle 开关**），每次出现文本完全一致
- 校验表固定 4 列：字段 / 必填 / 校验规则 / 校验失败提示；校验规则多条用分号（；）分隔
- 跨模块规则集中在第四章；模块内只引用不重复展开
- Mermaid 节点文字 ≡ 正文术语（同一概念不允许两种说法）
- 优先级只出现 P0 / P1；P2 统一塞进第七章未来规划
- 字段命名遵守 `../templates/default-style/field-naming.md`
- Mermaid 样式遵守 `../templates/default-style/mermaid-styles.md`

## 硬规则：Ask first（必须向用户确认）

- 新功能点归属哪个模块 / 是否新起模块
- 是否要同步更新第四章跨模块规则
- 原型图是否已存在（有则给路径；无则占位 `./assets/{name}.jpg`）
- MVP 不做的能力具体是哪几项（用于生成 "不支持 XX" 小节）
- 目标用户画像 / 核心业务实体 / 与已有模块的依赖关系（新建模块必问）

## 硬规则：Never do（红线，命中即失败）

- ❌「约束与边界」写入**业务边界**或**数据影响**以外的任何内容
- ❌ 为对称补写 "数据影响：无" / "业务边界：无" —— 整个小节一起删，不保留空槽
- ❌ 创建 X 与编辑 X 合并在同一段落或共用同一张字段表（必须各自独立一张表）
- ❌ MVP 文档中出现 P2 优先级
- ❌ Mermaid 节点与正文对同一概念使用两种说法
- ❌ 把 MVP 不做的能力藏在段落中间或塞进约束与边界（必须独立 "**不支持 XX**" 小节）
- ❌ 列表 / 表格类组件缺空状态文案
- ❌ 用绝对路径引用素材（图片一律 `./assets/xxx.jpg`）
- ❌ 字段只写规则不给中文提示文案（提示文案用中文直引号 `"..."` 包裹）

## 约束与边界的维度去处（不得放入该小节）

| 维度 | 正确位置 |
|---|---|
| 前置条件 | 「触发方式」的子 bullet |
| 交互规则 | 功能点的平级 bullet（与触发方式同级） |
| 展示规则 | 功能点的平级 bullet（主要用于页面型 / 搜索） |
| 外部依赖 | 第七章 · 外部依赖表 |
| 容错规则 | 第七章 · 主要风险表的"应对措施"列 |
| MVP 不做的能力 | 独立的 "**不支持 XX**" 小节 |
| 功能定位说明 | 放回模块概述或页面概述 |

## 交付前自检

详见 `prd-quality-checklist.md`。本范式特有的 grep 级自检：

- [ ] 7 章齐全且顺序正确（模块级）
- [ ] 无空槽占位（"数据影响：无"、"业务边界：无"）
- [ ] 无 P2 优先级出现在 MVP 文档
- [ ] 字段表全部 4 列
- [ ] 每个列表 / 表格组件紧跟"空状态"段落
- [ ] MVP 不做的能力全部独立 "**不支持 XX**" 小节
- [ ] 专有名词在全文每次出现形式完全一致
- [ ] 所有 mermaid 节点文字能在正文中找到同名术语
- [ ] 所有内部引用（"详见 XXX"）指向实际存在的小节 / 规则编号
- [ ] 字段顺序在"功能清单 / 架构图 / 功能详述 / 原型图"之间完全一致
- [ ] 所有图片引用都是 `./assets/` 相对路径
- [ ] 跨模块规则在第四章出现；模块内所有提及点均为引用而非重复展开

## 不在范围内

- 技术方案 / API 契约设计 → 另起 tech-spec，不是 PRD
- 运营文案 / 发布物 → 另起 marketing-doc
- 已存在的 v1 PRD 历史迁移 → 人工确认后再动，本范式不自动化
- 埋点方案 / 数据指标定义 → 另起 analytics-spec
